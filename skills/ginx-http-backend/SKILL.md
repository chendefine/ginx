# ginx HTTP Backend

## Follow the development flow

1. Read the target repository's `AGENTS.md` and local development instructions.
2. Inspect `go.mod`, route registration, and generation inputs before editing:

   ```bash
   rg --files -g 'go.mod' -g 'oapi-ginx.y*ml' -g 'openapi.y*ml' -g '*.gen.go'
   rg -n 'github.com/chendefine/ginx|Code generated by oapi-ginx|Register.*Routes'
   ```

3. Choose the starting point, but finish every server through the ginx runtime:

   - Prefer the contract-first path for new or changed APIs: read [references/codegen.md](references/codegen.md), generate the Go protocol types and scaffolding, then read [references/runtime.md](references/runtime.md) and implement the actual business behavior.
   - If `oapi-ginx.yaml` or generated oapi-ginx files already exist, always read both references. Regenerate first, implement `ServerInterface` in handwritten files, and run it through a configured `ginx.Engine`.
   - Use the runtime-only path for services that intentionally do not use OpenAPI, or for isolated handwritten routes outside the generated contract.

4. Treat the checked-out ginx source and its documentation as authoritative when working inside this repository. Read `README.md` for the runtime path and `README_CODEGEN.md` for the codegen path. Open `docs/RUNTIME_REFERENCE.md` or `docs/CODEGEN_REFERENCE.md` only for behavior not covered by the main path.

## Preserve the core contract

- Treat code generation as an intermediate contract/scaffolding step, not as a complete API implementation. It does not generate business logic.
- Implement every generated `ServerInterface` method in handwritten Go code and pass that implementation to the generated `RegisterRoutes` function.
- Implement ordinary business handlers as `func(context.Context, *Req) (*Rsp, error)`.
- Keep `*gin.Context` at the HTTP boundary; retrieve it with `ginx.GinContext(ctx)` only when required.
- Configure a dedicated `ginx.Engine` before registering routes. Treat route configuration as an immutable registration-time snapshot.
- Enable `ginx.WithStrictJSONBody(true)` and mask ordinary internal errors for public services unless the project's compatibility contract says otherwise.
- Return expected business failures with `ginx.Error(code, message).Status(httpStatus)`.
- Change OpenAPI specs, generator config, or templates instead of hand-editing files marked `Code generated by oapi-ginx; DO NOT EDIT.`

## Execute safely

1. Prefer this end-to-end sequence for contract-driven services:

   ```text
   OpenAPI spec -> oapi-ginx -> generated Go contracts/scaffolding -> handwritten business implementation -> ginx runtime
   ```

2. Make the smallest change at the correct ownership layer: OpenAPI spec/config, generator/template source, or handwritten service/runtime code.
3. Preserve existing response envelopes, status codes, validation rules, and generated API names unless the requested contract changes them.
4. Regenerate before implementing or repairing generated interfaces. Never stop at successful generation when the task requires a working API.
5. Format changed Go files and run the target repository's tests.
6. In the ginx repository, run:

   ```bash
   go test ./... -count=1
   ```
